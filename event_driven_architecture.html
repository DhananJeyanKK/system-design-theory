Event-Driven Architecture · Foundations
Why Events? — The Motivation
Synchronous request chains force users to wait for every downstream call. One slow service poisons the entire response. Events break that chain.

The three participants
📤
Producer
Emits events when something happens. Doesn't know who listens. Doesn't wait for a response. Fully decoupled from consumers.
📡
Message Broker
Receives, stores, and delivers events. Decouples producers from consumers in time and space. Acts as the nervous system of the architecture.
📥
Consumer
Subscribes to topics or queues. Processes events independently at its own pace. Multiple consumers can handle the same event differently.

Key differences
Aspect Request / Response Event-Driven
CommunicationSynchronous, blockingAsynchronous, non-blocking
CouplingTight — caller knows calleeLoose — producer unknown to consumer
ControlCaller drives the flowInversion of control — broker routes
Failure handlingOne failure = whole chain failsConsumers fail independently
ScalingAll services scale togetherEach consumer scales independently
Reality: Use both
No system is purely one or the other. Request/Response for immediate user-facing data (product listings, search). Event-driven for background processing, cross-service coordination, and anything that doesn't need an instant reply.

Message Delivery Semantics
Messages can be lost at the producer, the broker, or the consumer. Delivery semantics define the guarantee — and every guarantee has a cost.

✗ Cons
Messages can be silently dropped. No retry on no-ACK. Data loss is accepted.

Publisher retries until ACK — duplicates possible
✓ Pros
No message is ever lost. Guaranteed delivery. Consumers process everything.
✗ Cons
Duplicate events possible. Consumers must be idempotent. Higher latency.
Use when
Push notifications, product reviews, emails. Missing one is worse than duplicate.
Exactly once — dedup + atomic commit
✓ Pros
Perfect guarantee. No duplicates. No drops. The holy grail of reliability.
✗ Cons
Highest latency. Not all brokers support it. Complex to implement.
Use when
Financial transactions, inventory deductions, billing events. Every event must happen exactly one time.
Design Patterns · Distributed Transactions
The Saga Pattern
Microservices lose ACID transactions — each service has its own DB. The Saga pattern provides distributed transaction management through compensating operations.
The Problem: Lost Atomicity
In a monolith, updating Order + Payment + Inventory in one DB transaction is atomic — it all succeeds or all rolls back. In microservices, each service has its own DB. A partial failure leaves the system in an inconsistent state with no built-in rollback. Saga solves this.
Orchestrator coordinates all services
How it works
A dedicated Workflow Orchestration Service calls each microservice in sequence. It tracks state and knows what to do when a step fails. On failure: it calls compensating operations to undo completed steps. The orchestrator is the single source of truth for the transaction state.
Services emit events, react to events
How it works
No central orchestrator. Each service listens for events and emits the next event when done. On failure: the failing service emits a compensating event that other services react to, undoing their work. More decoupled — but harder to visualize the overall flow.

Design Patterns · Read/Write Separation
CQRS — Command & Query Responsibility Segregation
Separate the model that updates data (Commands) from the model that reads data (Queries). Optimize each independently. Scale each independently.
Command Side (Writes)
INSERT, UPDATE, DELETE operations. Optimized for consistency and write throughput. Single source of truth. When a command completes successfully, it publishes an event to the broker.
Query Side (Reads)
READ operations only. Denormalized, pre-joined views optimized for query patterns. Can have multiple specialized read models (one for listing, one for analytics). Eventually consistent.
Interactive CQRS — write flows through broker to read replica
Select an operation
Key Trade-off
Eventual consistency — read model lags slightly behind write model. Not ACID. For most use-cases, sub-second lag is acceptable.
Scalability Win
Scale read and write services independently. High read traffic? Add read replicas. High write load? Scale command service only.
Cross-DB Joins
With DB-per-service, joins are impossible. Create a dedicated read model that combines data from multiple services — a pre-joined view updated by events.
Design Patterns · Audit & History
Event Sourcing
Instead of storing the current state, store every event that led to that state. Replay events to reconstruct state at any point in time.
Traditional State Storage ❌
You store: balance = $450. But how did it get there? No history. Can't audit. Can't reconstruct the past. When a bug corrupts state — it's gone.
Event Sourcing ✓
You store: deposited $500 → withdrew $50. The balance is derived. Complete audit trail. Replay to reconstruct any past state.

The power combo: Event Sourcing + CQRS
Write fast → Event log → Query service reads in-memory
Complete audit history
Every state change is recorded. Regulatory compliance, debugging, root-cause analysis — all trivial.
Fast writes
Appending to an event log is the fastest write operation possible. No reads before writes. No update contention.
Snapshot optimization
Instead of replaying 10,000 events, take periodic snapshots. Replay only from the latest snapshot — massive performance gain.
