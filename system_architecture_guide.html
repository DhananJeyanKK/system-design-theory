🌐 DNS (Domain Name System)

What is DNS?

DNS is the phonebook of the internet. It translates human-readable domain names (like www.example.com) into IP addresses (like 192.0.2.1) that computers use to identify each other on the network.

How DNS Works

  1. User enters URL: Browser needs to find the IP address
  2. DNS Resolver Query: Request sent to DNS resolver (usually ISP's DNS server)
  3. Root Server: Resolver queries root nameserver for TLD info
  4. TLD Server: Queries Top-Level Domain server (.com, .org, etc.)
  5. Authoritative Server: Gets actual IP address from domain's nameserver
  6. Response: IP address returned to browser
  7. Connection: Browser connects to the web server

DNS Record Types

  • A Record: Maps domain to IPv4 address (example.com → 192.0.2.1)
  • AAAA Record: Maps domain to IPv6 address
  • CNAME Record: Alias of one domain to another (www.example.com → example.com)
  • MX Record: Mail exchange servers for email
  • TXT Record: Text information, often for verification
  • NS Record: Nameserver records indicating authoritative DNS servers

Pros and Cons of DNS

✅ Pros

  • Human-Friendly: Easy to remember names instead of IP addresses
  • Flexibility: Can change server IPs without changing domain
  • Load Distribution: One domain can resolve to multiple IPs
  • Caching: Reduces lookup time through caching at multiple levels
  • Decentralized: Distributed system is resilient
  • Global Reach: Works across the entire internet

❌ Cons

  • Propagation Delay: Changes can take 24-48 hours to propagate globally
  • Single Point of Failure: If DNS fails, website is unreachable even if server is up
  • Security Vulnerabilities: DNS spoofing, cache poisoning, DDoS attacks
  • Privacy Concerns: DNS queries can reveal browsing habits
  • No Built-in Security: Original DNS had no encryption (DNSSEC adds this)
  • Complexity: Troubleshooting DNS issues can be difficult

DNS Caching

DNS responses are cached at multiple levels to improve performance:

  • Browser Cache: Temporary storage in your browser
  • OS Cache: Operating system level cache
  • Router Cache: Local network router cache
  • ISP Cache: Internet Service Provider DNS cache

TTL (Time To Live): Each DNS record has a TTL value (in seconds) that specifies how long it should be cached.

⚖️ Load Balancing

What is Load Balancing?

Load Balancing distributes incoming network traffic across multiple servers to ensure no single server bears too much load. This improves application responsiveness and availability.

Load Balancing Algorithms

1. Round Robin

Distributes requests sequentially across all servers. Simple and effective when all servers have similar capacity.

2. Least Connections

Sends requests to the server with the fewest active connections. Good for varying request processing times.

3. Weighted Round Robin

Similar to round robin but servers with higher weights receive more requests. Useful when servers have different capacities.

4. IP Hash

Uses client IP address to determine which server receives the request. Ensures same client always goes to same server.

5. Least Response Time

Sends requests to server with lowest response time and fewest connections.

6. Random

Randomly selects a server for each request. Simple but less predictable.

Types of Load Balancers

Layer 4 (Transport Layer) Load Balancing

  • Operates at TCP/UDP level
  • Faster, less resource intensive
  • Routes based on IP address and port
  • Cannot inspect packet contents

Layer 7 (Application Layer) Load Balancing

  • Operates at HTTP/HTTPS level
  • Can route based on URL, headers, cookies
  • Enables content-based routing
  • More resource intensive but more flexible

Pros and Cons of Load Balancing

✅ Pros

  • High Availability: If one server fails, traffic routes to others
  • Scalability: Easy to add/remove servers as demand changes
  • Performance: Distributes load prevents server overload
  • Flexibility: Can route traffic based on various criteria
  • Maintenance: Can take servers offline without downtime
  • Geographic Distribution: Route users to nearest server

❌ Cons

  • Single Point of Failure: Load balancer itself can fail (mitigated with redundancy)
  • Complexity: Adds another layer to manage and troubleshoot
  • Cost: Hardware/software costs for load balancers
  • Session Persistence: Maintaining user sessions across servers can be tricky
  • SSL Termination: Can become a bottleneck for encrypted traffic
  • Configuration: Incorrect configuration can cause issues

Health Checks

Load balancers continuously monitor server health to avoid routing traffic to failed servers:

  • Active Health Checks: Periodically sends requests to check server status
  • Passive Health Checks: Monitors actual traffic and detects failures
  • HTTP Health Check: Sends HTTP request and expects specific response code
  • TCP Health Check: Attempts to establish TCP connection

🌍 GSLB (Global Server Load Balancing)

What is GSLB?

GSLB (Global Server Load Balancing) distributes traffic across geographically dispersed data centers. It routes users to the nearest or best-performing data center, improving performance and providing disaster recovery.

GSLB operates at the DNS level, returning different IP addresses based on user location, server health, and load.

How GSLB Works

  1. User makes DNS request: User queries example.com
  2. GSLB receives query: DNS request reaches GSLB controller
  3. Evaluation: GSLB evaluates:
    • User's geographic location
    • Data center health and availability
    • Current load on each data center
    • Network latency to each location
  4. Decision: GSLB selects optimal data center
  5. Response: Returns IP address of chosen data center
  6. Connection: User connects to selected data center

GSLB Routing Methods

1. Geographic Proximity

Routes users to the nearest data center based on geographic location.

2. Performance-Based

Routes based on actual measured latency and response times.

3. Load-Based

Distributes traffic based on current server load across data centers.

4. Active-Passive Failover

Primary data center handles all traffic; backup takes over if primary fails.

5. Active-Active

All data centers actively serve traffic simultaneously.

6. Weighted Round Robin

Distributes traffic based on configured weights for each data center.

Pros and Cons of GSLB

✅ Pros

  • Improved Performance: Users connect to nearest/fastest data center
  • Disaster Recovery: Automatic failover if data center goes down
  • Global Reach: Serve users worldwide efficiently
  • Compliance: Keep data in specific geographic regions
  • Cost Optimization: Balance load to avoid expensive peak pricing
  • Reduced Latency: Minimizes network hops and distance

❌ Cons

  • Complexity: Requires sophisticated infrastructure and monitoring
  • Cost: Multiple data centers are expensive to maintain
  • DNS Caching: DNS TTL can delay failover and routing changes
  • Accuracy: IP geolocation isn't always accurate (VPNs, proxies)
  • Data Synchronization: Keeping data consistent across regions is challenging
  • Configuration: Complex setup and maintenance

GSLB vs Standard Load Balancing

Aspect Standard Load Balancing GSLB
Scope Single data center Multiple data centers globally
Layer Layer 4 or Layer 7 DNS layer (Layer 3)
Decision Based On Server load, connections Geography, latency, health
Failover Time Milliseconds to seconds Seconds to minutes (DNS TTL)
Use Case High traffic to single location Global user base

📨 Message Brokers

What is a Message Broker?

A Message Broker is middleware that enables applications to communicate by sending and receiving messages. It decouples services by acting as an intermediary that receives messages from producers and delivers them to consumers.

Popular Message Brokers: RabbitMQ, Apache Kafka, AWS SQS, Azure Service Bus, Redis Pub/Sub

Message Broker Patterns

1. Point-to-Point (Queue)

One producer sends message to a queue. One consumer receives and processes it. Message is removed from queue after consumption.

2. Publish-Subscribe (Pub/Sub)

Producer publishes message to a topic. Multiple subscribers can receive the same message. Each subscriber gets a copy.

3. Request-Reply

Sender sends message and waits for response. Receiver processes and sends reply back.

4. Topic-Based

Messages are published to topics. Consumers subscribe to specific topics of interest.

Synchronous vs Asynchronous Operations

Synchronous Communication (Without Message Broker)

// Client makes HTTP request response = httpClient.post("/api/process", data); // Client WAITS for response // Blocked until server responds print(response);

Characteristics:

  • Client waits for immediate response
  • Tightly coupled - both services must be available
  • Simple to understand and debug
  • Server failure causes client failure

Asynchronous Communication (With Message Broker)

// Producer sends message to queue broker.send("processing-queue", data); // Producer continues immediately, doesn't wait // Message is processed later by consumer print("Message sent, continuing..."); // Separate consumer process consumer.onMessage((message) => { processData(message); });

Characteristics:

  • Producer sends and continues immediately
  • Loosely coupled - services can be down temporarily
  • Better scalability and resilience
  • More complex to track and debug

When to Use Message Brokers

✅ Use Message Brokers When:

  • Decoupling Services: Services don't need to know about each other
  • Asynchronous Processing: Tasks can be processed later (email sending, image processing)
  • Load Leveling: Smooth out traffic spikes by queuing requests
  • Reliability: Messages persist even if consumer is down
  • Microservices: Enable event-driven architecture
  • Multiple Consumers: Different services need same data (pub/sub)
  • Rate Limiting: Control processing rate

❌ Don't Use Message Brokers When:

  • Immediate Response Needed: User waiting for result (e.g., login)
  • Simple Direct Communication: Two services with straightforward interaction
  • Low Message Volume: Overhead not justified
  • Tight Consistency Required: Need immediate confirmation of state changes

Pros and Cons of Message Brokers

✅ Pros

  • Decoupling: Services are independent
  • Scalability: Easy to add more consumers
  • Reliability: Messages persist, retry on failure
  • Load Leveling: Smooth out traffic spikes
  • Asynchronous: Non-blocking operations
  • Multiple Consumers: Broadcast to many services
  • Fault Tolerance: Continues working if one service fails

❌ Cons

  • Complexity: Additional infrastructure to manage
  • Latency: Not suitable for real-time responses
  • Message Order: Can be difficult to guarantee ordering
  • Debugging: Harder to trace message flow
  • Duplicate Messages: May need idempotency handling
  • Storage: Messages consume disk space
  • Learning Curve: Team needs to understand messaging patterns

Real-World Use Cases

E-commerce Order Processing

Order placed → Message to: Inventory Service, Payment Service, Shipping Service, Notification Service (all process independently)

Video Processing

User uploads video → Queue message → Worker processes (transcoding, thumbnails, metadata) without blocking user

Logging & Monitoring

Applications send logs to message broker → Multiple consumers: Storage, Analytics, Alerting

Microservices Events

User updates profile → Event published → Multiple services react (email, analytics, recommendations)

🚪 API Gateway

What is an API Gateway?

An API Gateway is a server that acts as a single entry point for all client requests to a backend system. It sits between clients and microservices, routing requests, enforcing policies, and providing cross-cutting concerns.

Popular Solutions: Kong, AWS API Gateway, Azure API Management, Apigee, Nginx, Traefik

Key Features of API Gateway

1. Request Routing

Routes incoming requests to appropriate microservice based on path, headers, or other criteria.

/api/users/* → User Service /api/orders/* → Order Service /api/products/* → Product Service

2. Authentication & Authorization

Validates API keys, JWT tokens, OAuth tokens before forwarding requests.

3. Rate Limiting

Prevents API abuse by limiting number of requests per client/time period.

4. Request/Response Transformation

Modifies requests before sending to backend or responses before returning to client.

5. Caching

Caches responses to reduce backend load and improve response times.

6. Load Balancing

Distributes requests across multiple instances of a service.

7. Protocol Translation

Converts between different protocols (HTTP to gRPC, REST to SOAP).

8. Monitoring & Logging

Centralized logging, metrics collection, and request tracing.

When to Use an API Gateway

✅ Use API Gateway When:

  • Microservices Architecture: Multiple backend services need unified access
  • Multiple Client Types: Mobile, web, IoT devices with different needs
  • Cross-Cutting Concerns: Authentication, logging, rate limiting needed everywhere
  • Legacy Integration: Need to expose old systems with modern API
  • API Versioning: Support multiple API versions simultaneously
  • Third-Party APIs: Provide APIs to external partners
  • Security: Hide internal service structure from public internet

❌ May Not Need API Gateway When:

  • Monolithic Application: Single backend service
  • Internal Only: Services only communicate internally
  • Simple Architecture: Few services with direct client access
  • Performance Critical: Extra hop adds unacceptable latency

How API Gateway is Used

Typical Request Flow:

  1. Client sends request: GET https://api.example.com/users/123
  2. Gateway receives request
  3. Authentication: Validates JWT token in header
  4. Rate limiting check: Ensures client hasn't exceeded limits
  5. Routing decision: Routes to User Service
  6. Request forwarding: Sends to http://user-service:8080/123
  7. Response processing: May add headers, transform data
  8. Caching: Stores response for future requests
  9. Logging: Records request details
  10. Return to client: Sends response back

API Gateway Patterns

Backend for Frontend (BFF)

Separate gateway for each client type (mobile, web, IoT), each optimized for its client.

Aggregation Pattern

Gateway calls multiple services and combines results into single response.

Client Request: GET /dashboard Gateway calls: - User Service (get user info) - Order Service (get recent orders) - Notification Service (get alerts) Gateway combines and returns all data in one response

Service Mesh vs API Gateway

  • API Gateway: North-south traffic (external → internal)
  • Service Mesh: East-west traffic (service → service)

Important Considerations

Performance

  • Gateway adds latency (typically 10-50ms)
  • Use caching to reduce backend calls
  • Monitor gateway resource usage

Availability

  • Gateway is critical - use redundancy
  • Deploy multiple instances with load balancer
  • Have failover strategy

Security

  • Gateway is security boundary
  • Implement DDoS protection
  • Use WAF (Web Application Firewall)
  • Keep gateway updated and patched

Complexity

  • Don't overload gateway with business logic
  • Keep routing rules simple
  • Document all gateway configurations

Pros and Cons of API Gateway

✅ Pros

  • Single Entry Point: Simplifies client interaction
  • Centralized Logic: Authentication, logging, rate limiting in one place
  • Protocol Translation: Support different protocols
  • Security: Hides internal architecture
  • Aggregation: Combine multiple service calls
  • Flexibility: Easy to change backend without affecting clients
  • Analytics: Centralized monitoring and metrics

❌ Cons

  • Single Point of Failure: If gateway down, all services unavailable
  • Latency: Additional network hop
  • Complexity: Another component to manage
  • Bottleneck: Can become performance bottleneck
  • Development Dependency: Changes may require gateway updates
  • Cost: Infrastructure and operational costs
