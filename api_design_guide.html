Categories of APIs

🌍
Public APIs

Definition: Open to external developers and third parties, often with minimal restrictions.

Use Cases:

  • Payment gateways (Stripe, PayPal)
  • Social media integration (Twitter, Facebook)
  • Mapping services (Google Maps)
  • Weather data (OpenWeatherMap)

Characteristics:

  • Extensive documentation required
  • Rate limiting and API keys
  • Stable versioning
  • High security standards
🔒
Internal/Private APIs

Definition: Used exclusively within an organization, not exposed externally.

Use Cases:

  • Microservices communication
  • Internal data sharing between teams
  • Backend-to-backend integration
  • Employee management systems

Characteristics:

  • Less formal documentation
  • Faster iteration cycles
  • Internal authentication only
  • Network-level restrictions
🤝
Partner APIs

Definition: Shared with specific business partners under agreements.

Use Cases:

  • B2B integrations
  • Supply chain management
  • White-label solutions
  • Reseller platforms

Characteristics:

  • Controlled access with contracts
  • Custom SLAs (Service Level Agreements)
  • Dedicated support
  • Enhanced security and monitoring

API Best Practices

📄 Pagination

Handle large datasets by breaking them into manageable chunks.

Common Pagination Methods:

  • Offset-based: ?limit=20&offset=40
  • Page-based: ?page=3&size=20
  • Cursor-based: ?cursor=eyJpZCI6MTAwfQ== (best for real-time data)
{ "data": [...], "pagination": { "total": 1000, "page": 2, "size": 20, "next": "/api/users?page=3&size=20", "prev": "/api/users?page=1&size=20" } }
🔄 Idempotency

Same request produces same result, even if called multiple times. Critical for reliability.

Idempotent Methods:

  • GET, PUT, DELETE: Naturally idempotent
  • POST: Needs idempotency keys
POST /api/payments Headers: Idempotency-Key: unique-uuid-12345 // First call: Creates payment // Repeated calls with same key: Returns same payment, no duplicate charge

Benefits: Prevents duplicate charges, safe retries, network fault tolerance

📦 Versioning

Manage API changes without breaking existing clients.

Versioning Strategies:

  • URI Versioning: /api/v1/users, /api/v2/users
  • Header Versioning: Accept: application/vnd.myapi.v2+json
  • Query Parameter: /api/users?version=2
// Old version (v1) GET /api/v1/users/123 Response: { "name": "John Doe" } // New version (v2) - added email GET /api/v2/users/123 Response: { "name": "John Doe", "email": "john@example.com" }
🎯 Complete Encapsulation

Hide internal implementation details; expose only necessary interfaces.

Principles:

  • Don't expose database structure in API responses
  • Use DTOs (Data Transfer Objects) to transform data
  • Abstract business logic behind clear endpoints
  • Version internal changes separately from API
// ❌ Bad: Exposes database columns { "user_id": 123, "created_at_timestamp": 1234567890, "is_deleted": 0 } // ✅ Good: Clean abstraction { "id": 123, "createdAt": "2024-01-15T10:30:00Z", "status": "active" }
⚡ Asynchronous Operations

Handle long-running tasks without blocking the client.

Pattern:

  1. Client initiates task → receives task ID immediately
  2. Server processes task in background
  3. Client polls status or receives webhook notification
// Step 1: Initiate async task POST /api/reports/generate Response: 202 Accepted { "taskId": "task-789", "status": "processing", "statusUrl": "/api/tasks/task-789" } // Step 2: Check status GET /api/tasks/task-789 Response: 200 OK { "taskId": "task-789", "status": "completed", "result": { "downloadUrl": "/api/reports/download/xyz" } }

Use for: File processing, batch operations, complex calculations, external API calls

RPC (Remote Procedure Call) Style

🎯 What is RPC?

RPC (Remote Procedure Call) style APIs focus on actions and procedures rather than resources. The API calls look like function calls.

Features of RPC

Action-Oriented

Endpoints represent operations/functions

POST /calculateTax
POST /sendEmail
Simple & Direct

Easy to understand for developers familiar with function calls

Flexible Parameters

Pass any data structure as function arguments

When to Use RPC Over REST

  • Action-heavy operations: When your API performs actions rather than managing resources (e.g., calculations, transformations)
  • Internal microservices: Communication between services where resources don't matter
  • Simple CRUD doesn't apply: Operations that don't fit REST verbs well
  • Real-time performance: When you need lightweight, fast calls (gRPC)

Advantages of RPC

  • ✅ Simplicity: Straightforward function-like calls
  • ✅ Performance: Binary protocols like gRPC are very fast
  • ✅ Strongly typed: With gRPC/Protocol Buffers, you get type safety
  • ✅ Language agnostic: Works across different programming languages
  • ✅ Bidirectional streaming: gRPC supports streaming data both ways

REST (Representational State Transfer) Style

🎯 What is REST?

REST is an architectural style that focuses on resources and uses standard HTTP methods to perform operations on those resources.

Features of REST

Resource-Based

Everything is a resource with a URI

/users/123
/products/456
HTTP Methods

Uses GET, POST, PUT, DELETE

Stateless

Each request contains all needed info

Cacheable

Responses can be cached for performance

Uniform Interface

Consistent, predictable structure

Self-Descriptive

Messages contain metadata

When to Use REST Over Other Styles

  • CRUD operations: When managing resources (Create, Read, Update, Delete)
  • Public APIs: REST is well-understood and documented
  • Web services: Works naturally with HTTP and web browsers
  • Caching needs: HTTP caching works out of the box
  • Discoverability: Resources can be explored through links (HATEOAS)

Advantages of REST

  • ✅ Scalability: Stateless nature allows easy scaling
  • ✅ Flexibility: Supports multiple data formats (JSON, XML)
  • ✅ Caching: Built-in HTTP caching improves performance
  • ✅ Simplicity: Uses familiar HTTP protocol
  • ✅ Visibility: Easy to monitor and debug
  • ✅ Standardization: Well-established conventions

Swagger / OpenAPI Specification

🔍 What is Swagger?

Swagger is now called OpenAPI Specification (OAS). It's a standard, language-agnostic way to describe REST APIs.

Key Components:

  • OpenAPI Specification: The standard format (YAML/JSON)
  • Swagger UI: Interactive API documentation
  • Swagger Editor: Tool to design APIs
  • Swagger Codegen: Generates client/server code

Where Swagger Fits in REST

Swagger is primarily designed for REST APIs. It describes:

  • Available endpoints (paths)
  • HTTP methods for each endpoint
  • Request/response formats
  • Authentication methods
  • Parameters and data models

Example: When you create a REST API, you write an OpenAPI spec, and Swagger UI automatically generates interactive documentation where developers can test your API directly in the browser.

Does Swagger Work with RPC?

Short answer: Not well.

Swagger/OpenAPI is designed around REST principles (resources, HTTP methods). For RPC-style APIs:

  • JSON-RPC: Can be documented with OpenAPI, but it's awkward since all calls use POST
  • gRPC: Uses Protocol Buffers, not OpenAPI. gRPC has its own tooling:
    • .proto files define the service
    • gRPC reflection for service discovery
    • Tools like grpcurl, BloomRPC for testing
