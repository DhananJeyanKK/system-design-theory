Strangler Fig Pattern

An interactive simulation of migrating a monolith to microservices — step by step, without a big-bang rewrite.

What is the Strangler Fig Pattern?

Named after the strangler fig tree 🌿 that grows around an existing tree, gradually replacing it. In software: you incrementally migrate functionality from a legacy monolith to new microservices — routing more traffic to the new services over time — until the monolith can be safely retired. No risky big-bang rewrite. Live traffic validates each step.

🛒 Real-world example — E-commerce platform

How a large e-commerce site might apply this pattern over 6–18 months

1
Add facade — Deploy an API Gateway (e.g. AWS API Gateway, NGINX) in front of the existing monolith. Zero changes to the monolith code. All traffic still flows through.
2
Extract User Auth — Build a standalone Auth Service. Update facade routing rules: /auth/* routes to the new service, everything else to monolith. Deploy, monitor, done.
3
Extract Product Catalog — Build a Catalog Service backed by its own database. Gradually shift /products/* traffic. Run both services in parallel during the cutover window.
4
Extract Orders & Payments — These are tightly coupled so migrate together. Use an event bus (Kafka/SQS) to sync order state between old and new systems during transition.
5
Retire the monolith — Once all domains are extracted and traffic is 0%, decommission the monolith. The strangler fig has completed the replacement.

Key principles

🔄 Incremental
Migrate one bounded context at a time. Never freeze development.
🛡️ Safe rollback
Route rules can flip back to monolith instantly if a new service misbehaves.
📊 Validated in prod
Real traffic tests each extracted service before the monolith code is deleted.
🧵 Co-existence
Old and new systems run simultaneously. The facade is the seam between them.
