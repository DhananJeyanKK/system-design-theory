Principles · Data Architecture
Database per Microservice
Each service owns its own data store. No shared schema. No cross-service DB joins. This is the price of true independence — and it's worth it.
Why shared DB breaks cohesion
❌ Shared Database Problem
Service A changes a column → Service B's queries break. Schema migrations require coordinating all teams simultaneously. Cohesion is violated — services are secretly coupled through the database.
✓ DB per Service Solution
Each service abstracts its own storage. Schema changes are local. You can version APIs without breaking consumers. Switch from Postgres to MongoDB? Only that one team needs to know.

Trade-offs
⚡ Added Latency
Service A needs Service B's data? Network call required. No local join. Design your data boundaries so cross-service calls are rare.
🔗 No SQL Joins
Cross-database joins are impossible. Data must be assembled at the application layer via API composition or event-driven denormalization.
⚗️ Distributed Atomicity
Transactions now span network calls. Traditional ACID across services is gone — use Saga pattern or eventual consistency. Patterns exist; plan for them.
Principles · Code Sharing
DRY doesn't apply the same way in Microservices
Don't Repeat Yourself is gospel in monoliths. In microservices, shared libraries become the hidden coupling that destroys independence.
Shared Library = Tight Coupling
Any bug or change in the shared lib forces every consumer to rebuild → retest → redeploy. You've recreated a monolith dependency graph — just spread across services.

Rebuild Retest Redeploy Vulnerability cascade
The Root Cause
Shared business logic between two microservices usually means incorrect boundaries. The fact that you need to share it is a signal — not a problem to work around, but a boundary to reconsider.
🔥 Dependency Hell
Shared Library v1.0 works. Then v1.1 drops. Service A upgrades. Service B can't upgrade yet (breaking change). Service C depends on A's internal behavior. Now you can't update anything without a full cascade. This is the monolith's stranglehold, distributed.
Situation 1: Shared Business Logic
Option 1 — Reevaluate Boundaries
If two services share business logic, their boundaries may be wrong. Merge them into one service or re-draw the boundary so the shared logic lives in exactly one place.
Option 2 — Create a Reusable Service
Extract the shared logic into its own microservice. Consumers call it via API. The service is independently deployable, versioned, and owned by one team.
Situation 2: Common Data Models for Communication
Option 1 — Shared Library ✓ (acceptable here)
For data transfer objects / API schemas only — no business logic — a shared library is acceptable. Changes to data models are versioned, not behavioral.
Option 2 — Schema + Code Generation
Define schemas in Protobuf / OpenAPI / Avro. Generate client code per language. Each service regenerates independently. No runtime coupling.
✓ Safe to Duplicate
Utility methods that change frequently — each service has its own optimized version and can evolve independently.
Generic helpers (date formatting, string utils) — duplication makes migration to a new language much easier.
Utility methods Format helpers Validation rules
✗ Don't Duplicate
Business rules that represent your domain — these belong in a service, not copy-pasted across 8 codebases. Bugs multiply.
Security / auth logic — one vulnerability becomes eight vulnerabilities.
Business rules Auth/security logic Payment calculations
Sidecar Pattern
Package the shared process as a separate deployable that runs on the same host as the service. Minimal performance overhead. No network call — it's local IPC.
Good sidecar candidates
Logging Retry logic Health checks TLS termination Metrics collection

These are stable cross-cutting concerns that rarely change and have no business logic.
Principles · Data Strategy
Data Duplication — costly or necessary?
Duplication seems wasteful. But for services that must respond immediately, caching another service's data is often the right trade-off — with one strict rule.

When duplication is justified
Service needs data from another service to serve requests immediately (display Product name in Order service). Calling out on every request adds unacceptable latency or creates a hard dependency on availability.

Immediate responses Reduce latency Avoid runtime dependency
The golden rule — one source of truth
Data duplication gives you eventual consistency — not strict consistency. The source of truth must live in exactly one service. All copies are derived and may lag.

If you need strict consistency: avoid duplication and call the owner service directly.
Event-driven synchronization
The owner service publishes events when data changes (e.g. product.price.updated). Consumers subscribe and update their local cache. This keeps copies fresh without tight coupling — the producer doesn't know who's listening.
Principles · Team Organization
Structured Autonomy
The myth: each team picks their own everything. The reality: complete autonomy creates infrastructure chaos, non-uniform APIs, and soaring costs. The answer is structured autonomy — freedom within boundaries.
❌ Full Autonomy Problems
High infra setup cost Ongoing maintenance overhead Steep learning curves Non-uniform APIs Security gaps Compliance nightmares
The Three-Tier Model — click each tier to expand
1
Tier 1 · Fully Restrictive
Everyone must follow — no exceptions
These are non-negotiable foundations. Deviating here causes org-wide chaos.

CI/CD pipeline standard Monitoring & alerting tooling Security standards Compliance requirements Public API design guidelines

Why restrictive? These affect every other team. An inconsistent CI/CD means ops can't support anyone. Inconsistent monitoring means on-call can't debug incidents.
2
Tier 2 · Freedom within Boundaries
Team choice — with guardrails
Teams pick from an approved list — not from the entire ecosystem.

Programming language Database technology Internal API style Test frameworks

Example: "You may use Go, Python, Java, or Node. You may use PostgreSQL, MongoDB, or Redis." The list is curated — infra team has pre-built support and expertise for these options.
3
Tier 3 · Complete Autonomy
Your call — total ownership
No oversight needed — teams own these decisions fully and bear the consequences.

Release schedule Internal scripts & tooling Code style & conventions Sprint cadence Local dev environment

Why total freedom? These decisions have zero impact on other teams. Team A deploying daily vs weekly affects only Team A.

Principles · Frontend Architecture
Micro-Frontends
Apply microservices thinking to the frontend. Split a monolithic SPA into independently deployable UI fragments — each owned by a full-stack team end-to-end.
Interactive — Monolith SPA vs Micro-Frontend
Independent Deployability
Teams release their UI fragment on their own schedule. The Checkout team ships without waiting for the Product team to finish their changes.
Easy to Test in Isolation
Each micro-frontend is a standalone SPA for testing purposes. No need to spin up the whole application to test the Basket fragment.
Full-Stack Ownership
One team owns the entire vertical slice: API, business logic, database, and UI for their domain. Conway's Law working for you, not against you.
